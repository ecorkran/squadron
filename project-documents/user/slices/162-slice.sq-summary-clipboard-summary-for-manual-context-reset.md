# Slice Design: /sq:summary вҖ” Clipboard Summary for Manual Context Reset

## Overview

Add a `/sq:summary [template]` slash command (plus a small supporting
`sq _summary-instructions` hidden CLI helper) that lets the user produce
a custom, template-driven summary of the *current interactive Claude
Code conversation* and copy it to the clipboard. The user can then
`/clear` and paste the summary into the fresh context, effectively
performing a deterministic, user-controlled "clear-with-custom-summary"
that `/compact [with instructions]` fails to provide reliably.

This is a **small, single-purpose slice**. It adds no new SDK session
plumbing, no new template formats, and no new config keys. It reuses
the compaction template machinery already built for slice 157
(PreCompact hook) and slice 158 (SDK session rotate): templates in
`src/squadron/data/compaction/`, `load_compaction_template()`,
`render_with_params()`, CF best-effort param gathering, and the
existing `compact.template` config key as the default.

The summary is generated by the **current Claude Code session itself**
вҖ” not by a squadron subprocess вҖ” because only the current session has
access to the conversation. Squadron's role is exclusively to (a)
resolve and render the template's instruction text, and (b) provide a
clipboard sink. Everything else is the slash command telling the
assistant what to do.

## Motivation

Today, when a user wants to continue a conversation in a fresh context
(to escape bloat, long tool outputs, or to focus on a new subtask),
they have two unattractive options:

1. **`/compact [with instructions]`** вҖ” non-deterministic. Sometimes
   honors custom instructions, sometimes ignores them and produces a
   default-shaped summary. User study by the project owner confirmed
   this is unreliable even for simple instructions like "keep only
   slice 158 context."

2. **`/clear`** вҖ” instant and deterministic but loses everything.

What users actually want is "`/clear` but with a custom, predictable
summary injected." Pipelines already solve this for pipeline-owned SDK
sessions (slice 158 session-rotate, slice 161 summary-step `emit:
[rotate]`). Interactive Claude Code sessions need the equivalent, but
squadron cannot directly manipulate the interactive session's context
вҖ” that's Claude Code's province.

The observation that unlocks this slice: squadron does not need to
manipulate the context. If it can produce good summary *instructions*
and give the user a clean path to get the resulting summary onto the
clipboard, the user can `/clear` and paste manually. That's two
keystrokes after `/sq:summary` runs, and the result is deterministic
because the current assistant generates the summary *in this session*
following the instructions вҖ” not during a `/compact` pass where the
instructions are merely a suggestion.

## Non-Goals

- **No full automation of `/clear`.** Claude Code does not (cleanly)
  expose a programmatic `/clear` to extensions. Manual `/clear` + paste
  is the intended user flow and is already painless.
- **No new SDK session or daemon plumbing.** The current Claude Code
  session is the summary target; squadron stays out of it.
- **No new template format or new template files.** Uses existing
  compaction templates (`default`, `minimal`, `minimal-sdk`) as-is.
- **No Windows clipboard support in this slice.** macOS `pbcopy` is
  the primary target; Linux `xclip`/`wl-copy` fallback is a freebie if
  present. Windows can be added later.
- **No summary-to-file, summary-to-stdout, or other emit destinations.**
  Clipboard-only. If users want file/stdout, that is what pipelines
  with `summary: { emit: [file|stdout] }` are for (slice 161).
- **No chat-visible dump of the summary.** The whole point is a clean
  handoff вҖ” the summary lives on the clipboard, not in the chat.

### PreCompact hook removal

The `_precompact-hook` CLI command (slice 157) is dead code вҖ” it was
dropped from `sq install-commands` because the PreCompact hook cannot
work reliably (non-deterministic instruction injection). Rather than
maintaining two consumers of shared helpers, this slice removes
`precompact_hook.py` entirely and salvages its useful logic (template
loading, CF param gathering, rendering) into the new
`summary_render.py` module. Cleanup includes the hook's test file,
`app.py` registration, and any install/settings references.

### Code Structure

**New file: `src/squadron/cli/commands/summary_instructions.py`**
- Typer command function `summary_instructions(template: str | None)`.
- Resolves template name via the precedence chain (arg вҶ’ config вҶ’
  `"minimal"`).
- Calls shared helpers (see below), prints rendered text to stdout.
- On failure (missing template, etc.), prints a clear error to stderr
  and exits 1. No JSON, no silent degradation.

**New file: `src/squadron/pipeline/summary_render.py`**
- Two pure functions salvaged from the removed `precompact_hook.py`:
  - `resolve_template_instructions(template_name, *, cwd) -> str` вҖ”
    template loading + CF param gathering + rendering. Raises
    `FileNotFoundError` on missing template.
  - `gather_cf_params(cwd) -> dict[str, object]` вҖ” best-effort CF
    param gather (`project`, `slice`, `phase`).

**Removed: `src/squadron/cli/commands/precompact_hook.py`**
- Dead code вҖ” no longer installed by `sq install-commands`.
- Useful logic salvaged into `summary_render.py` above.
- Also removes: test file, `app.py` registration, install/settings
  references.

**Modified: `src/squadron/cli/app.py`**
- One new registration line:
  `app.command("_summary-instructions", hidden=True)(summary_instructions)`
- Name prefixed with `_` and `hidden=True` following the
  `_precompact-hook` convention вҖ” this is an internal helper, not a
  user-facing CLI surface.

**New file: `commands/sq/summary.md`**
- Slash command markdown, following the pattern in
  `commands/sq/run.md` etc.
- Parses `$ARGUMENTS` for an optional template name (first word).
- Instructs the assistant to perform the four-step flow above.

### Precedence rules

Template name resolution, in order:
1. Explicit `--template X` / slash-command arg.
2. `compact.template` config value (via `get_config`), if set.
3. Hardcoded default `"minimal"`.

The chain **reuses** the existing `compact.template` config key on
purpose. "What template should I use when I want to deterministically
summarize my current context" is the same question whether the
trigger is a PreCompact hook, a pipeline compact step, or a
`/sq:summary` invocation. One knob, three callers. No new config.

Param rendering precedence is unchanged from existing behavior:
`LenientDict` leaves unknown placeholders intact so that
templates referencing `{slice}` or `{phase}` don't blow up in
projects without CF.

### Clipboard portability

The clipboard write happens in the slash command's Bash step, not in
Python. This keeps the CLI pure-text-to-stdout and avoids adding a
clipboard dependency to `sq`.

Shell chain (in the slash command's Bash instructions):

```
pbcopy 2>/dev/null || \
  xclip -selection clipboard 2>/dev/null || \
  wl-copy 2>/dev/null || \
  { echo "no clipboard tool found (install xclip or wl-clipboard)" >&2; exit 1; }
```

- macOS: `pbcopy` always present, always wins.
- Linux X11: `xclip` fallback.
- Linux Wayland: `wl-copy` fallback.
- Windows: falls through to error message. User instructed to wait
  for a future slice or install one of the above.

`pyperclip` is already in squadron's deps (slice 161), so if the
portability chain ever needs to move into Python, we can switch
without adding a dependency. For this slice, keeping clipboard in the
shell is simpler and easier to debug when users report "it didn't
copy."

## Data Flow

### Happy path (macOS, user has CF project with `slice=162`)

1. User types `/sq:summary minimal-sdk` in Claude Code.
2. Slash command expands to the prompt from `commands/sq/summary.md`,
   with `$ARGUMENTS=minimal-sdk`.
3. Assistant runs Bash: `sq _summary-instructions --template minimal-sdk`.
4. `summary_instructions` CLI:
   - Resolves `minimal-sdk` (arg wins, no config lookup).
   - Loads `src/squadron/data/compaction/minimal-sdk.yaml`.
   - Gathers CF params: `{project: "squadron", slice: "162-slice...", phase: "..."}`.
   - Renders `{slice}`, `{keep_section}` placeholders.
   - Prints rendered text to stdout.
5. Assistant receives the rendered instructions as Bash output.
6. Assistant generates a summary of the current conversation following
   those instructions, as plain text (no preface, no explanation).
7. Assistant pipes the summary to the clipboard shell chain.
8. Assistant prints one-line confirmation:
   `Summary copied to clipboard (2,147 chars, template: minimal-sdk).`

### Default template path (no argument)

Step 3 becomes `sq _summary-instructions` (no flag). The CLI resolves
to `compact.template` config value (if set) or `"minimal"`. Everything
else unchanged.

### Missing template

`sq _summary-instructions --template bogus` prints
`Error: compaction template 'bogus' not found. Searched: ~/.config/squadron/compaction, <builtin>/compaction`
to stderr and exits 1. The slash command detects the non-zero exit
and surfaces the error to the user as its final message вҖ” no summary
generated, no clipboard write attempted.

### No clipboard tool

The clipboard shell chain exits 1 with "no clipboard tool found ..."
on stderr. The assistant sees the failure, reports it to the user,
and does not claim success. The generated summary is discarded (not
dumped to chat вҖ” that would defeat the point).

## Interfaces

### `sq _summary-instructions` CLI

Hidden command; not listed in `sq --help`.

```
sq _summary-instructions [--template NAME]
```

- `--template NAME` вҖ” optional. Template name (without `.yaml`
  extension). Overrides config default.
- `--cwd PATH` вҖ” hidden, follows `_precompact-hook` precedent; defaults
  to `.`. Used for CF param lookup.

**Output:** rendered template instructions to stdout. Plain text.
No surrounding structure.

**Exit codes:**
- `0` вҖ” success.
- `1` вҖ” template not found or render failure. Error message on stderr.

### `/sq:summary` slash command

```
/sq:summary [template]
```

- `template` вҖ” optional. Template name. If omitted, uses
  `compact.template` config value or `"minimal"`.

**User-visible output:** a single confirmation line on success, or an
error message on failure. The summary itself lives on the clipboard.

## Success Criteria

1. Running `/sq:summary` in Claude Code (any squadron project) produces
   a summary of the current conversation on the clipboard, with a
   one-line confirmation in chat. No summary text appears in chat.
2. Running `/sq:summary minimal-sdk` uses the `minimal-sdk` template
   instead of the config default.
3. Running `/sq:summary bogus-template` surfaces a clear error (not a
   fake success, not silent), and nothing ends up on the clipboard.
4. `sq _summary-instructions` (direct CLI invocation) prints rendered
   template text to stdout and exits 0.
5. `sq _summary-instructions --template bogus` prints an error to
   stderr and exits 1.
6. The existing `_precompact-hook` command continues to behave
   identically after the shared-helper extraction (verified by existing
   tests + new tests on the extracted helpers).
7. macOS `pbcopy` is the primary path. On a Linux box with `xclip` or
   `wl-copy` installed, the fallback chain works. On a box with none,
   the user gets a clear "no clipboard tool found" error instead of a
   silent failure.
8. `sq install-commands` picks up the new `commands/sq/summary.md`
   and installs it alongside existing `/sq:*` commands, so a fresh
   squadron install includes `/sq:summary` without extra steps.

## Risks

**Low overall.** The slice reuses battle-tested machinery and adds
minimal new code. Two specific concerns:

1. **Slash command must not dump the summary into chat.** The whole
   user benefit disappears if the summary appears in the chat before
   it gets to the clipboard вҖ” that's exactly the bloat the user is
   trying to escape. The slash command's instructions must be explicit
   and unambiguous: clipboard only, one-line confirmation only. A test
   for this is hard to write automatically (it's an instruction-
   following concern), so the verification walkthrough explicitly
   checks for it.

2. **Linux clipboard tool absence.** A Linux user without `xclip` or
   `wl-copy` gets a clean error, not a silent failure. But the first
   time they encounter it, they have to stop and install a package.
   This is acceptable for a 1/5 effort slice; documented in the
   slash command's error message.

## Open Questions

1. **Shared helper module name.** Proposed `pipeline/summary_render.py`
   because the helpers are closely related to existing
   `pipeline/compact_render.py` (where `LenientDict` lives). Alternative:
   put them in `pipeline/compact_render.py` directly, since that file
   is already about "rendering compaction instructions." Decision can
   be made in implementation вҖ” either is fine.

2. **Default confirmation format.** Proposed
   `Summary copied to clipboard (N chars, template: T).` Could include
   the resolved template's description for extra clarity. Leaving
   minimal for now; can be tuned during implementation.

3. **Should `/sq:summary` without a template argument print the
   resolved template name before generating?** E.g. "Using template:
   minimal-sdk (from config)". Possibly useful for transparency;
   possibly noise. Leave out for v1; add if users ask.

## Effort Estimate

**1/5** вҖ” small, tight slice. Most of the code is boilerplate:
register a new hidden CLI command, write a slash command markdown file,
extract two functions into a helper module. The conceptual work is
minimal because all the hard parts (template loading, rendering, CF
param gathering, `LenientDict`) already exist.
